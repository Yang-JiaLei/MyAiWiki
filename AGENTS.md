# AGENTS

这份文件定义本 Vault 的 Agent 协作规则。

目标不是临时问答，而是把知识逐步沉淀成一个持续维护的 Obsidian Wiki。参考模式来自 Karpathy 的 LLM Wiki，并适配当前这套中文目录结构。

## 协作角色

- 你负责提供资料、提出问题、判断什么值得长期保留。
- Agent 负责阅读、整理、归档、交叉链接、更新索引和记录维护日志。
- 正式知识优先沉淀到页面，而不是停留在对话里。

## 本库三层职责

### `个人存货`

- 放你自己的原生内容。
- 包括项目资料、学习笔记、技能树、思考碎片、Prompt 资产。
- 不承载外部知识的正式百科结论。

### `Agent产出`

- 放过程材料。
- 包括抓取结果、研究中间稿、聊天记录、Wiki 草稿、系统日志。
- 只有当内容稳定、可复用时，才迁入 `个人存货` 或 `Wiki百科`。

### `Wiki百科`

- 放外部知识的长期沉淀。
- 采用 `实体 / 主题 / 原始来源 / 运维` 四个层次。
- 任何正式结论都尽量能追溯到来源页。

## Agent 默认工作流

### 1. 摄入新资料

按以下顺序执行：

1. 判断资料属于哪一层。
2. 如果是外部资料，先在 `Wiki百科/3. 原始来源/` 下创建或更新来源页。
3. 抽取实体、主题、关键主张、证据、待验证点。
4. 优先更新已有实体页或主题页，不轻易创建平行页。
5. 必要时新建正式页，并补上双向链接。
6. 更新 [[Wiki百科/0. 运维/来源台账|来源台账]]。
7. 在 [[log]] 追加一次维护记录。

### 2. 回答问题

按以下顺序检索：

1. 先读 [[index]] 或相关 README 找入口。
2. 再读相关正式页。
3. 需要时参考来源页或过程稿。
4. 如果答案有长期价值，应回写成正式页面或补充到既有页面。

### 3. Lint 与整理

定期检查：

- 断链
- 重复页
- 孤立页
- 旧结论是否被新来源推翻
- 过程稿是否应迁移或归档
- README 与入口页是否还能正确导航

## 写作规则

- 优先更新，不优先新建。
- 不把整篇原文复制进正式 Wiki。
- 事实、推断、假设分开写。
- 未核实内容必须标记不确定性。
- 页面只承担一个主职责，不混写来源登记、个人记录和正式结论。
- 每次改动尽量补至少一条有意义的内部链接。

## 页面放置规则

- 你自己写的长期内容：`个人存货/`
- AI 研究过程与草稿：`Agent产出/`
- 外部资料摘要与百科沉淀：`Wiki百科/`
- 规则、模板、lint、台账：`Wiki百科/0. 运维/`

## Frontmatter 约定

沿用 [[Wiki百科/0. 运维/页面Schema|页面Schema]]：

```yaml
---
title:
created:
updated:
type:
category:
tags: []
aliases: []
source_status:
confidence:
---
```

常用 `type`：

- `index`
- `entity`
- `topic`
- `source`
- `research-question`
- `rule`
- `workflow`
- `schema`
- `ledger`
- `report`
- `standard`

## 命名规则

- 保持现有中文目录结构，不做英文化迁移。
- 文件名优先使用稳定、可读、可检索的中文命名。
- 同一对象如果已有页面，继续沿用旧命名，不另起近义页面。

## CLI 与工具约定

- 优先使用 Obsidian CLI、Obsidian Markdown 语法和现有 README 导航。
- Dataview、Bases、Canvas 可作为展示层，不作为唯一事实来源。
- 当前机器上 `obsidian help` 已可在用户终端执行；若 Agent shell 未继承 PATH，可显式补入 `D:\\Obsidian` 后再调用。

## 本对话的默认协作方式

- 你给我资料、主题、问题或整理目标。
- 我优先检查现有页面，再决定更新还是新建。
- 每次处理后，我会告诉你：
  - 改了哪些页面
  - 为什么这么放
  - 后面还缺什么
