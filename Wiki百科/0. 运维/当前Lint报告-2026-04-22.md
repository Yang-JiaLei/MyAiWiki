# 当前Lint报告 2026-04-22

## 结论

本轮 lint 已从“结构收口”进入“入口修复 + schema 收敛 + 弱连接治理”阶段。

当前最值得优先处理的不是大规模迁移，而是：

1. 修复顶层入口断链
2. 补齐 frontmatter 缺口
3. 让正式实体页和主题页逐步收敛到当前页面 schema

## 工具说明

- 本轮使用 `obsidian-cli` 作为工作流参考
- 但在当前 Agent 进程里，`obsidian help` 类命令的输出不会稳定回传
- 因此这次 lint 的主统计来自文件扫描，不依赖 CLI stdout
- 用户终端侧的 CLI 本身可用，只是输出回流方式与当前会话不一致

## P1 问题

### 1. 顶层入口存在真实断链

当前已确认以下入口目标不存在：

- `[[待办任务]]`
- `[[日志]]`
- `[[知识库总览]]`
- `[[个人存货/5. Prompt库/README]]`

主要命中位置：

- `主页.md`
- `个人存货/README.md`
- `Agent产出/3. Skills/安装日志.md`
- `Wiki百科/0. 运维/Agent配置规范.md`
- `个人存货/1. 项目/1.4 Agent 配置.md`
- 多个实体页和主题页中的 `[[日志.md#最近操作]]`

影响：

- 首页导航会把人带到不存在的页面
- 一批正式页仍引用旧入口命名，导致“可读但不可跳转”

建议处理：

1. 确认 `待办任务 / 日志 / 知识库总览` 是否要恢复为真实页面
2. 如果不恢复，则统一改为现有入口：`index / log / 主页`
3. 检查 `个人存货/5. Prompt库/` 是否目录缺失或已重命名

### 2. 正式 Wiki 页面没有真正收敛到当前页面 schema

按 `页面Schema.md` 的现行要求扫描后，绝大多数正式实体页和主题页仍缺少必备章节。

典型情况：

- 实体页普遍缺少：
  - `## 是什么`
  - `## 关键事实`
  - `## 相关主题`
  - `## 相关来源`
  - `## 待验证点`
- 主题页普遍缺少：
  - `## 定义`
  - `## 核心观点`
  - `## 关键方法或机制`
  - `## 相关实体`
  - `## 相关来源`
  - `## 开放问题`

代表页面：

- `Wiki百科/1. 实体/1.1 人物/andrej-karpathy.md`
- `Wiki百科/1. 实体/1.2 公司/openai.md`
- `Wiki百科/1. 实体/1.4 工具/gemini-cli.md`
- `Wiki百科/2. 主题/2.1 AI/RAG.md`
- `Wiki百科/2. 主题/2.2 技术/LLM.md`
- `Wiki百科/2. 主题/2.4 领域知识/Gemini.md`

影响：

- 当前 schema 已存在，但还没有成为正式写作习惯
- 后续摄入会持续把新内容堆到旧结构里，导致 lint 永远无法收敛

建议处理：

1. 先选 `entity` 和 `topic` 各 2 到 3 页做模板化改写
2. 确认新模板可用后，再按目录批量收敛

## P2 问题

### 3. metadata 质量不一致

本轮扫描发现：

- 27 个 Markdown 页缺少 `created`
- 6 个 Markdown 页完全没有 frontmatter
- 3 个页缺少 `type / category / updated`

其中最值得优先处理的是这批无 frontmatter 的来源文件：

- `Wiki百科/3. 原始来源/3.1 文章/baoyu-skills-CHANGELOG.md`
- `Wiki百科/3. 原始来源/3.1 文章/baoyu-skills-CLAUDE.md`
- `Wiki百科/3. 原始来源/3.1 文章/baoyu-skills-README.md`
- `Wiki百科/3. 原始来源/3.1 文章/gemini-cli-README.md`
- `Wiki百科/3. 原始来源/3.1 文章/prompts-chat-PROMPTS.md`
- `个人存货/2. 学习笔记/2.2 参考文档/构建总结.md`

说明：

- 部分 `missing-created` 来自 README 类入口页，风险低于正式内容页
- 但无 frontmatter 的来源文件会绕开当前 schema 和 Dataview 体系，优先级更高

### 4. orphan / deadend 数量偏高，但需要区分真实问题和噪音

本轮结果：

- orphan：34
- deadend：78

其中一部分属于可接受噪音：

- README 入口页天然可能没有入链或出链闭环
- 原始来源中的导入文档本来就可能只作为登记材料存在
- 运维规范页不一定需要大量出链

但以下类型更值得处理：

- 正式实体页和主题页被判定为 orphan
- 研究资料和技能资料缺少回链
- 部分项目页和学习笔记与总入口没有形成稳定导航关系

建议处理：

1. 先治理正式 Wiki 页的 inbound/outbound
2. 再治理 `Agent产出` 中有长期价值但缺回链的过程稿

## P3 问题

### 5. 旧命名和旧入口仍残留在文档内部

本轮仍发现大量旧式链接习惯，例如：

- `[[日志.md#最近操作]]`
- `[[知识库总览]]`
- 旧路径和乱码路径的残留示例

说明：

- 一部分来自旧文档正文
- 一部分来自历史模板或样例内容
- 需要在“真实链接”与“示例文本”之间做区分，不应盲目批量替换

## 本轮建议处理顺序

1. 修复首页及关键 README 的断链入口
2. 为无 frontmatter 的来源页补最小 metadata
3. 选 2 到 3 个实体页、2 到 3 个主题页做 schema 样板收敛
4. 再做第二轮 lint，观察 orphan 和 deadend 是否显著下降
